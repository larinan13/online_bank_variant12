# Контрольный вопрос №12

**Студент:** Ларина А.Е.  
**Группа:** 454  
**Формулировка вопроса:** Какие аномалии модификации могут возникнуть, если оставить таблицу во второй нормальной форме? Приведите гипотетический пример на основе вашей схемы.

---

## Развёрнутый ответ

### Суть второй нормальной формы (2НФ)

Таблица находится во второй нормальной форме, если она уже находится в первой нормальной форме (1НФ) и все неключевые атрибуты полностью зависят от всего первичного ключа (отсутствуют частичные зависимости). Проблемы начинаются, когда в таблице есть составной первичный ключ, и некоторые атрибуты зависят только от части этого ключа.

### Какие аномалии возникают в 2НФ?

Оставляя таблицу во 2НФ, мы сталкиваемся с тремя типами аномалий модификации:

1. **Аномалия вставки** — невозможно добавить данные о сущности, если у неё ещё нет связи с другой сущностью
2. **Аномалия обновления** — изменение значения требует модификации нескольких строк, что ведёт к противоречиям
3. **Аномалия удаления** — при удалении связи теряется информация о самой сущности

### Гипотетический пример из моей схемы банка

В моей базе данных `online_bank_variant12` есть таблица `bookings` (записи на обслуживание). Предположим, что я **нарушила вторую нормальную форму** и создала таблицу `employee_bookings` с составным первичным ключом:

```sql
-- АНТИПАТТЕРН: таблица во 2НФ (содержит частичные зависимости)
CREATE TABLE employee_bookings (
    employee_id INT,
    booking_date DATE,
    booking_time TIME,
    employee_last_name VARCHAR(50),      -- зависит ТОЛЬКО от employee_id
    employee_department VARCHAR(100),    -- зависит ТОЛЬКО от employee_id
    client_id INT,                       -- зависит от всего ключа
    amount DECIMAL(15,2),                -- зависит от всего ключа
    PRIMARY KEY (employee_id, booking_date, booking_time)
);
```
Здесь составной ключ: (employee_id, booking_date, booking_time).
Атрибуты employee_last_name и employee_department зависят только от employee_id (часть ключа) — это частичная зависимость, нарушение 2НФ.

### Аномалия вставки

Невозможно добавить нового сотрудника в систему, пока у него нет ни одной записи на обслуживание:

```sql
-- Хотим добавить нового сотрудника, но не можем, потому что
-- для таблицы employee_bookings нужен составной ключ
INSERT INTO employee_bookings (employee_id, employee_last_name, employee_department)
VALUES (10, 'Новый сотрудник', 'Новый отдел');
-- Ошибка: NULL в полях booking_date, booking_time, входящих в PRIMARY KEY
```

### Аномалия обновления

Если сотрудник переводится в другой отдел, пришлось бы обновлять множество записей:

```sql
-- Сотрудник с id=1 переводится из 'Отдел физ.лиц' в 'Депозитный отдел'
-- Приходится обновлять ВСЕ строки с этим сотрудником
UPDATE employee_bookings 
SET employee_department = 'Депозитный отдел' 
WHERE employee_id = 1;  -- Обновляет 15 строк
```

Если обновить не все строки (например, из-за ошибки приложения), возникнет противоречивость данных — один сотрудник будет числиться в разных отделах.

### Аномалия удаления

При удалении последней записи сотрудника теряется информация о самом сотруднике:

```sql
-- Удаляем последнюю запись сотрудника с id=3
DELETE FROM employee_bookings 
WHERE employee_id = 3 AND booking_date = '2026-05-02';
-- После этого в таблице не остаётся строк с employee_id=3
-- Информация о фамилии и отделе сотрудника потеряна НАВСЕГДА
```
### Как исправляет третья нормальная форма?

В моей правильной схеме информация о сотрудниках вынесена в отдельную таблицу employees:

```sql
-- Правильное решение (3НФ)
CREATE TABLE employees (
    employee_id INT PRIMARY KEY,
    last_name VARCHAR(50) NOT NULL,
    department VARCHAR(100) NOT NULL
);

CREATE TABLE bookings (
    booking_id INT PRIMARY KEY,
    employee_id INT NOT NULL,
    booking_date DATE NOT NULL,
    booking_time TIME NOT NULL,
    FOREIGN KEY (employee_id) REFERENCES employees(employee_id)
);
```

## Заключение

Оставление таблицы во второй нормальной форме создаёт серьёзные риски целостности данных. Переход к третьей нормальной форме (3НФ) через декомпозицию таблиц устраняет аномалии вставки, обновления и удаления, что критически важно для банковских систем, где данные о сотрудниках и клиентах должны быть непротиворечивыми.
